## 2. Findings

### 2.1 Run stopped by 504 on first batch analysis

- **What:** First batch of “do these issues still exist?” (50 issues, 188,956 chars prompt, claude-sonnet-4.5 via ElizaCloud) was sent at 06:10:27. At 06:13:36 (~3 min 9 s) the client logged `504/gateway timeout or request timeout, retrying` then after retry: `Batch analysis failed: 504 An error occurred with your deployment`.
- **Why it matters:** Batch analysis is the gate before the fix loop. Once it throws, the run exits and no fixes are attempted.
- **Likely cause:** ElizaCloud/gateway or deployment timeout on a large request (~47k tokens input + long output). 3 min is often above gateway timeouts (e.g. 60–120 s).
- **Recommendations:**
  - Reduce batch size for “issue existence” when using ElizaCloud (e.g. cap at 25 or 30 issues per batch) so each request stays under gateway limits.
  - Or: on 504 for batch analysis, retry with a smaller batch (e.g. half) and document that 504 can occur on large batches.

### 2.2 CodeRabbit startup wait (~2.5 min)

- **What:** After posting `@coderabbitai review`, the run waited and polled PR status until it decided to continue (logs show repeated “PR status fetched” with `botsWithEyesReaction: ["coderabbitai[bot]"]`, then “CodeRabbit startup check → reviewedCurrentCommit: false” and continuation).
- **Why it matters:** This is the **startup** “trigger then wait” path in `workflow/startup.ts`, not the post-push “wait for bot re-review” path. The recent `--no-wait-bot` / default no-wait only affects the **post-push** wait. So this run still did a long startup wait.
- **Recommendation:** If the goal is “don’t block on CodeRabbit,” consider also shortening or making optional the startup wait after triggering (e.g. skip wait or cap at 30–60 s and proceed with current comments).

### 2.3 LLM dedup behavior

- **What:** 38 files with 2+ issues each; one LLM call per file (cheap model for 2–4 issues, full model for 5+). Most responses were “NONE” (4 chars). A few had short GROUP lines (e.g. 24 chars). Claude sometimes returned reasoning plus “NONE”; parser correctly treated as no groups. Two additional merges were applied (fix-loop-utils, push-iteration-loop).
- **Verdict:** Dedup prompts and parsing behaved as intended. No change needed for correctness; reducing or capping concurrency is only for rate limits.

### 2.4 Prompts.log and slugging

- **What:** PROMPT #0001–#0077 and matching RESPONSE entries are in prompts.log with slugs like `#0001/llm-elizacloud`. output.log references the same slugs, so cross-file lookup works.
- **Verdict:** Dual logging and slug linkage are working and useful for debugging (e.g. jump from “504 on batch” to prompt #0077).

## 3. Recommendations (prioritized)

1. **Batch analysis 504:** Reduce batch size for issue-existence checks when provider is ElizaCloud (or on first 504, retry with half the issues). This unblocks the run on this PR.
2. **Startup CodeRabbit wait:** If “don’t block on CodeRabbit” should apply at startup too, add an option or cap to skip/shorten the wait after `@coderabbitai review`.
3. **Document 504 on large batches:** In docs or CHANGELOG, note that very large batch analysis requests can hit gateway/deployment timeouts (504) and that reducing `--max-context` or batch size may help.
