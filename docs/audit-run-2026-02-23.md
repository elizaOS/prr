## 3. Issues and observations

### 3.1 Fixer vs verifier disagreement (high impact)

- **Pattern:** Codex repeatedly returned `RESULT: ALREADY_FIXED` for:
  - **getTokenPair bypasses rate-limiting queue** — Fixer claimed `metadataEnqueue` is used and there is no `Promise.all`/`getMetadataWithRetry`; verifier said "getTokenPair still calls getMetadataWithRetry directly via Promise.all".
  - **Dead per-queue processor functions** — Fixer claimed code already addressed; verifier said the three functions are still present and unused.
- **Effect:** Multiple batch runs and single-issue focus attempts with no code changes, then rotation. Wastes fixer calls and time.
- **Possible causes:** Fixer may be reasoning over an outdated or partial snippet; or the prompt’s "current code" may not match the file on disk (e.g. after a prior partial fix). Verifier receives the actual diff/current code and is correct.
- **Recommendation:** When the fixer returns ALREADY_FIXED but verifier says still exists, consider (a) re-fetching the file content for the next attempt so the fixer sees the same state as the verifier, or (b) adding a one-line hint in the retry prompt: "Verifier reports the issue still exists at [citation]; re-check the current file content."

### 3.2 .prr/ in bot prediction and prompts (medium)

- **In prompts.log:** `.prr/lessons.md` appears in diff summaries (e.g. `diff --git a/.prr/lessons.md`) and in bot-prediction output as a suggested new bot concern ("FILE: .prr/lessons.md", "add to .gitignore").
- **In output.log:** Bot prediction listed ".prr/lessons.md — Auto-generated... consider adding to .gitignore" as likely new feedback. That is expected from the diff; no fix was requested for .prr/ in the fix loop (no comment on .prr/ in the queue).
- **Status:** If issue-analysis already excludes comments on paths under `.prr/` (tool-artifact filter), then no fix prompts target .prr/. Bot prediction will still mention .prr/ when it appears in the diff; optional improvement is to strip or mask `.prr/` from the diff (or from predicted feedback) so bots are not encouraged to comment on it.

### 3.3 Repeated global lessons in every prompt (low)

- **Observation:** The same 5 global lessons appear in every fix prompt (e.g. "Direct LLM ALREADY_FIXED for package.json" x3, "Direct LLM ALREADY_FIXED for tsconfig.build.json", "Fixer made no edits..."). Same for file-specific lessons (e.g. build.ts: "Fixer made no edits...").
- **Impact:** Token and clutter; the fixer sees the same lines every time even when the current issues are about service.ts only.
- **Recommendation:** Already improved by showing file-only lessons in AAR; in prompts, consider limiting global lessons to a small number or to those relevant to the current issue set (e.g. by file overlap).

### 3.4 Dismissal comment "already exists" (low)

- **Observation:** Many "Dismissal comment already exists" for the same path/line (e.g. `src/service.ts` line 158, `.prr/lessons.md` line 39, `build.ts` line 46) across multiple LLM calls during the dismissal-comment phase.
- **Impact:** Extra LLM calls that immediately skip; suggests multiple dismissed issues map to the same file/line and each triggers a "generate + check exists" flow.
- **Recommendation:** Before generating a dismissal comment, group by (filePath, line) and only generate once per location; or short-circuit when a comment at that location already exists.

### 3.5 Remaining issue: build.ts watch mode (design)

- **Issue:** "Dev watch mode never activates" — `bun --watch run build.ts` consumes `--watch` so `process.argv` never sees it; `watchRequested` (from env) is used in code but the verifier stated it’s always false because the script doesn’t receive the flag.
- **What was tried:** Direct LLM removed `watch: watchRequested`; verifier correctly said "removing watch entirely doesn’t fix the root cause".
- **Conclusion:** The remaining issue is genuinely hard (runtime flag vs env vs file watcher). Handoff prompt and AAR correctly surface it for human attention.
